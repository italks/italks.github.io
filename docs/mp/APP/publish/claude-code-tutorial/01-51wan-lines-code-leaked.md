## 一个失误，51万行代码曝光

2026年3月31日，一条推文引爆了技术圈。

安全研究员 Chaofan Shou（@Fried_rice）发现：Anthropic 发布的 npm 包 `@anthropic-ai/claude-code@2.1.88` 中，意外包含了 `.map` 文件。

这个文件有什么问题？

**59.8MB 的 `cli.js.map`，可以被逆向还原成完整的 TypeScript 源码。**

还原后：
- **1906 个文件**
- **51.2 万行 TypeScript 代码**
- **Claude Code 的完整实现逻辑**

这不是黑客攻击，而是 Anthropic 自己的失误。

这不是技术能力问题，而是工程流程问题——CI/CD 流水线没有过滤 `.map` 文件。

---

## 泄露原因：一个 `.map` 文件的低级失误

什么是 `.map` 文件？

这是 JavaScript 打包时生成的 **source map 文件**，用于将压缩后的代码映射回原始源码，方便调试。

正常情况下，生产环境的 `.map` 文件不应该发布到 npm。

但 Anthropic 的 CI/CD 流程，没有过滤这个文件。

结果：

```
cli.js.map（59.8MB）
    ↓ 逆向工具还原
完整 TypeScript 源码（1906个文件，51.2万行）
```

**一个文件配置的疏忽，暴露了整个项目的核心资产。**

## 对普通用户的影响：你的数据安全吗？

很多人担心：泄露的源码会不会暴露用户数据？

**答案：不会。**

泄露的是 **工具本身的代码**，不是用户数据。

Claude Code 的设计是：
- 用户数据存储在本地（`~/.claude/` 目录）
- API Key 存储在系统钥匙串
- 源码泄露 ≠ 数据泄露

但这件事提醒我们：

**AI 工具正在成为新的攻击面。**

## 从这次泄露中，我们能学到什么？

### 1. CI/CD 流程必须过滤敏感文件

`.map` 文件、`.env` 文件、密钥文件……这些都不应该进入发布包。

### 2. 定期审计发布内容

发布前检查 npm 包里有什么，而不是信任 CI/CD 配置。

### 3. 敏感信息不要硬编码

API Key、密钥、内部配置，应该用环境变量或密钥管理服务。
