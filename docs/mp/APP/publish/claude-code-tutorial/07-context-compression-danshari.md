# 上下文压缩：AI 是怎么"断舍离"的？

> 阅读时长：5分钟
>
> 51 万行代码教会我的信息筛选艺术。

## 对话太长，AI 开始"失忆"怎么办？

你和 AI 聊了很久，积累了大量对话：

- 修改了多次代码
- 遇到了几个错误
- 讨论了多种方案

突然，AI 开始"失忆"：

```
你：还记得第 3 步的方案吗？
AI：不好意思，我没有看到您之前提到的第 3 步方案。
```

**原因：上下文窗口接近限制，旧对话被压缩了。**

但压缩不是"删除"，而是"断舍离"——保留重要的，舍弃冗余的。

---

## Claude Code 的 9 段式压缩策略

从泄露的源码中，我们看到了结构化的压缩逻辑：

### 源码实现

```typescript
interface ConversationSummary {
  userIntent: string;        // 用户的原始意图
  errorRecords: ErrorLog[];  // 错误修复记录
  pendingTasks: Task[];      // 待办任务
  codeChanges: ChangeLog[];  // 项目结构变更
  decisionsMade: Decision[]; // 已做出的决策
  assumptions: string[];     // 当前假设
  constraints: string[];     // 已知约束
  contextFiles: string[];    // 已读取的文件
  recentInteractions: string; // 最近交互摘要
}
```

**关键：不是简单截断，而是结构化提取。**

---

## 保留什么？9 个关键字段

### 1. 用户原始意图（userIntent）

**保留内容：**

```
用户想要实现一个搜索功能，支持模糊匹配和高亮显示。
```

**为什么保留：**

AI 需要记住"最终目标"，才能在后续对话中保持方向一致。

### 2. 错误修复记录（errorRecords）

**保留内容：**

```
错误 1：TypeError - Cannot read property 'map' of undefined
修复：添加空数组默认值

错误 2：ESLint - 'useState' is not defined
修复：添加 import { useState } from 'react'
```

**为什么保留：**

避免重复犯同样的错误，节省调试时间。

### 3. 待办任务（pendingTasks）

**保留内容：**

```
- [ ] 添加单元测试
- [ ] 处理分页逻辑
- [ ] 优化搜索性能
```

**为什么保留：**

AI 知道还有哪些工作要做，不会遗漏。

### 4. 项目结构变更（codeChanges）

**保留内容：**

```
新增文件：
- src/components/SearchBar.tsx
- src/hooks/useSearch.ts

修改文件：
- src/App.tsx：添加搜索路由
```

**为什么保留：**

AI 知道项目当前状态，不会建议已存在的文件。

### 5. 已做出的决策（decisionsMade）

**保留内容：**

```
决策 1：使用 Zustand 而非 Redux，理由是项目规模较小
决策 2：搜索功能放在前端，而非后端 API
```

**为什么保留：**

避免 AI 重复提出已被否决的方案。

### 6. 当前假设（assumptions）

**保留内容：**

```
- 用户已安装 Node.js 18+
- 数据源是本地 JSON 文件
- 不需要用户登录功能
```

**为什么保留：**

AI 知道当前约束，不会建议不符合实际的方案。

### 7. 已知约束（constraints）

**保留内容：**

```
- 不使用 class 组件，只用函数式组件
- 样式使用 Tailwind，不用 CSS Modules
- API 请求必须带错误处理
```

**为什么保留：**

AI 生成的代码符合项目规范。

### 8. 已读取的文件（contextFiles）

**保留内容：**

```
已读取：
- src/App.tsx
- src/components/ProductList.tsx
- src/api/products.ts
```

**为什么保留：**

AI 知道哪些文件已了解，避免重复读取。

### 9. 最近交互摘要（recentInteractions）

**保留内容：**

```
最近 5 轮对话：
1. 用户：添加搜索功能
2. AI：创建 SearchBar.tsx
3. 用户：搜索没反应
4. AI：修复事件绑定问题
5. 用户：添加防抖
```

**为什么保留：**

保留最近的对话上下文，保持连贯性。

## 舍弃什么？三类信息

### 1. 已解决的细节

```
错误：xxx
修复：yyy

错误已解决 → 不需要保留详细堆栈信息
```

### 2. 重复信息

```
用户：帮我写个搜索功能
AI：好的，我来实现...

用户：搜索功能还没好吗？
AI：正在实现...

用户：搜索什么时候能好？
AI：马上就好...

→ 三次重复询问，压缩后只保留一次
```

### 3. 无关闲聊

```
用户：今天天气不错
AI：是的，适合写代码

用户：你觉得 TypeScript 好用吗？
AI：TypeScript 有很多优点...

→ 与当前任务无关的对话，会被舍弃
```

## 压缩不是删除：结构化摘要 vs 粗暴截断

### 粗暴截断

```
[对话历史] ... [截断] ... [最近对话]
```

问题：丢失中间的重要信息。

### 结构化摘要

```
[用户意图] [错误记录] [待办任务] ... [最近对话]
```

优势：关键信息以结构化形式保留。

**类比：**

- 粗暴截断 = 把旧的日记本扔掉
- 结构化摘要 = 把日记本整理成摘要，保留重点

## 对普通人的启发：如何和 AI 进行长对话

### 1. 主动总结关键信息

不要指望 AI 记住所有细节，关键信息要主动重复：

```
你：记住，我们用的是 React + TypeScript + Vite 技术栈。
```

### 2. 分段处理大任务

长任务拆成多个短对话：

```
对话 1：搭建基础结构
对话 2：实现核心功能
对话 3：优化和测试
```

### 3. 写下来比说出来好

把重要信息写成文件：

```
你：我在 CLAUDE.md 里写了技术栈和约束，请读取。
```

文件不会被压缩，始终可读。

### 4. 定期"重启"对话

对话太长时，主动开启新对话：

```
你：我们重新开始一个对话，我继续之前的工作。
```

把当前进度总结一下，作为新对话的起点。

## 总结

**上下文压缩 = 结构化的"断舍离"**

保留的 9 类信息：

1. 用户原始意图
2. 错误修复记录
3. 待办任务
4. 项目结构变更
5. 已做出的决策
6. 当前假设
7. 已知约束
8. 已读取的文件
9. 最近交互摘要

**核心原则：**

- 不是简单删除，而是结构化提取
- 保留影响后续决策的信息
- 舍弃已解决、重复、无关的内容

**对普通人的启发：**

- 主动总结关键信息
- 长任务分段处理
- 重要信息写成文件
- 定期"重启"对话
