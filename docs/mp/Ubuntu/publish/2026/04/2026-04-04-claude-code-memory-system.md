# 三层记忆系统：AI的"长期记忆"为什么比你想的重要

> 阅读时长：约 10 分钟
> 适合读者：使用 AI 编程工具有一定经验的开发者
> 前置知识：理解"上下文窗口"的基本概念

---

## 一个每天都在上演的困境

"上次我问了 AI 关于这段代码的问题，给了很长的背景介绍，结果今天开新对话，它又什么都不记得了。"

这是每个 AI 编程工具用户都遇到过的问题。

你花了 20 分钟解释项目背景，AI 终于理解了你的代码结构。第二天开新对话，发现又要重来一遍。

大多数人的反应是：**忍着，重新说一遍。**

少数人开始怀疑：**AI 工具是不是都这样？有没有解决办法？**

Claude Code 源码泄露之后，我们第一次看到了工业级解决方案——**三层记忆系统**。

这是整个 Claude Code 架构中最核心的设计，也是我认为对普通用户最有启发性的部分。

---

## 先说原理，再讲应用

为了让你真正用好这套系统，先花一点时间说清楚它是怎么工作的。

### 上下文窗口的天花板

AI 模型有上下文窗口限制——你可以理解为 AI 的"大脑容量"。

Claude Opus 4 的上下文窗口是 20 万 Token，大约相当于一部《战争与和平》的长度。

听起来很大，但实际使用中：

- 你的项目代码可能有 50 万行
- 历史对话记录可能有 10 万 Token
- 技术文档可能有 5 万 Token

全部塞进去？早就超了。

**大多数 AI 工具的选择是：超了就截断，从最近的对话继续。**

这意味着：对话早期提到的背景信息，会随着对话进行逐渐被"遗忘"。

### 粗暴解法 vs 智能解法

**粗暴解法**：扩大上下文窗口。

上下文不够 → 用更大窗口的模型 → 成本翻倍 → 速度变慢 → 还是不够用。

这是一个没有终点的军备竞赛。

**Claude Code 的解法**：不是扩大窗口，而是**建索引**。

## 三层记忆系统详解

Claude Code 的记忆系统分三层，每层承担不同的职能：

```
┌─────────────────────────────────────────────────────┐
│ 第一层：MEMORY.md          "书签" │ 常驻上下文      │
├─────────────────────────────────────────────────────┤
│ 第二层：CLAUDE.md          "手册" │ 按需加载        │
├─────────────────────────────────────────────────────┤
│ 第三层：会话上下文         "工作记忆"│ 动态变化      │
└─────────────────────────────────────────────────────┘
```

### 第一层：MEMORY.md——常驻书签

这是最薄的一层，也是**唯一始终存在于 AI 上下文中的记忆层**。

格式上，它是一个 Markdown 文件，每一行是一个"记忆条目"：

```
## 用户偏好
- 代码风格：2空格缩进，不用Tab
- commit信息：中文，简洁，50字内
- 测试要求：必须包含边界条件测试

## 项目注意事项
- payments模块是高风险区域，修改前先告知
- 老代码用jQuery，不要改成Vue
```

**关键约束：每行不超过 150 字符。**

这不是随便定的限制。Claude Code 的实现里，第一层的所有内容必须能塞进上下文窗口——所以必须精简。

**这层记忆的职能**：
- 存最重要的用户偏好（"他喜欢什么风格的代码"）
- 存项目级注意事项（"哪个模块是高风险的"）
- 充当其他记忆文件的"索引"（"详细内容见 `memdir/code-style.md`"）

**类比**：
> 第一层记忆就像你桌上的便签——永远在那儿，写的是最重要的提醒，字数有限，要精简。

### 第二层：CLAUDE.md——项目手册

这一层是开发者**主动维护的项目文档**。

和 MEMORY.md 的区别在于：MEMORY.md 是"给 AI 看的个人提醒"，CLAUDE.md 是"这个项目的完整使用说明书"。

CLAUDE.md 由开发者编写，通常包含：

```markdown
# 本项目说明

## 技术栈
- 后端：Node.js + Express
- 数据库：PostgreSQL + Prisma ORM
- 前端：Vue 3 + Vite
- 测试：Vitest

## 代码组织
- `/src/controllers` 路由处理
- `/src/services` 业务逻辑
- `/src/models` 数据库模型

## 已知坑
- 第3方支付回调有时序问题，重试逻辑见 utils/payment-retry.ts
- 旧API版本(v1)不再维护，但因客户原因暂不能删除

## 约定
- 所有API响应格式见 docs/api-response-format.md
- 错误码定义在 constants/error-codes.ts
```

**CLAUDE.md 什么时候被读取？**
- 新建 Claude Code 对话时
- 执行 `/restart` 重置会话时
- 用户主动要求 AI 读取项目背景时

**CLAUDE.md 不在第一层，是因为它内容可能很长，不应该常驻上下文**——只在需要的时候加载。

**类比**：
> 第二层记忆就像公司的内部 wiki——新人入职第一天要看，内容详细但不是随时都在眼前。

### 第三层：会话上下文——工作记忆

这一层就是普通对话里的上下文。

AI 在当前对话中收到的所有消息、生成的所有回复，都在第三层。

当第三层内容超过上下文窗口时，Claude Code 会触发**自动压缩**——把早期的对话内容压缩成结构化摘要，只保留关键信息。

压缩后的内容类似：
```
## 早期背景
- 用户在做一个订单系统，用的是NestJS框架
- 已解决：数据库连接池配置问题
- 已解决：用户认证中间件逻辑错误
- 进行中：支付模块接入，第3方回调处理有问题

## 待办
- 继续处理支付回调的时序问题
```

**类比**：
> 第三层记忆就像人类的工作记忆——当前正在处理的事，放在"桌面"上，关掉对话就没了。

## 为什么这个设计值得普通用户学习

这个三层系统不是 Claude Code 独有的设计理念，它代表的是**"AI 协作的正确姿势"**。

### 普通人的问题：把所有希望寄托在"上下文足够大"

大多数人在用 AI 时，会试图在一次对话里塞进所有信息：
- 粘贴 200 行代码
- 写 500 字背景描述
- 期待 AI 从中理解一切

结果：上下文爆炸，AI 开始"失忆"，输出质量断崖式下降。

三层记忆系统给我们的启示是：**与其一次说太多，不如建立分层的信息供给机制。**

### 核心心法：让 AI 养成"查阅习惯"

Claude Code 的三层记忆系统有一个关键特性：**MEMORY.md 的内容是"始终在"的。**

这意味着，当你每次新建对话，AI 看到的第一件事就是 MEMORY.md——你的偏好、你的注意事项，全部在那里。

**这相当于让 AI 形成了一个条件反射：每次对话开始，先读 MEMORY.md。**

**普通用户可以这样做：**

在你常用的 AI 编程工具里（不限 Claude Code），建立你自己的 MEMORY.md：

```markdown
# 我的编程助手记忆

## 我的偏好
- 代码风格：清晰胜过技巧，优先写可读性高的代码
- 注释要求：复杂的业务逻辑必须写中文注释
- 重构习惯：重构前先解释我的改动思路

## 常用技术栈
- 主要做：后端 Node.js + 前端 React
- 熟悉但不精通：Python 脚本、Shell 脚本
- 不熟悉的领域：iOS原生开发、Kubernetes运维

## 我的工作习惯
- 习惯git commit频率：每完成一个小功能就提交
- 不喜欢：一次性生成大段代码，更喜欢分步生成
- 调试偏好：遇到Bug先看日志，不要直接猜测原因
```

这个文件放在你项目的根目录，或者维护在一个固定位置。

**关键点**：每次新对话，先告诉 AI："先读一下我的 MEMORY.md"，让它形成习惯。

## 更高级的用法：利用 CLAUDE.md 做项目传承

CLAUDE.md 最强大的用法，是让它成为**项目知识传承的载体**。

一个团队中，新人接手老项目最痛苦的是什么？

- 代码里找不到注释解释为什么要这样设计
- 没有人告诉他为什么要引入那个第三方库
- 踩了一个坑之后，下一个新人又踩了一遍

CLAUDE.md 可以解决这个问题：

```markdown
# 项目知识传承文档

## 为什么我们用 Prisma 而不是 TypeORM？
- Prisma 的类型生成更完善，IDE 支持更好
- 当时调研结论见：docs/2024-06-db-orm-comparison.md

## 为什么订单状态机要自己实现不用状态机库？
- 当时的状态机库对嵌套状态支持不好
- 参考：GitHub Issue #234（已在文档库归档）

## 过去踩过的坑
- 2024-03: 支付回调并发问题，解决方案是加分布式锁
- 2024-07: 图片上传大小限制没配导致线上事故
```

把这些信息显式地记录下来，比藏在代码 review 记录里强一百倍。

## 源码给我们的工程启发

三层记忆系统的实现本身，也值得开发者思考：

### 1. 记忆触发是有策略的，不是每次都全量加载

Claude Code 不是每次对话都把 MEMORY.md 和 CLAUDE.md 全量塞进上下文。

它有一个**触发策略**：MEMORY.md 常驻，CLAUDE.md 只在新建对话时加载，历史对话中的重要信息压缩后保留。

**工程启发**：这不是随便设计的。背后有成本和性能的综合考量——全量加载记忆意味着每次都要多消耗 Token。

### 2. 记忆的写入是自动化的，不是靠用户手动维护

Claude Code 有一个 `autoDream` 机制：当满足某些条件时（比如对话轮次够了，或者检测到了值得记录的信息），AI 会在后台自动整理 MEMORY.md，把新学到的东西写进去。

**工程启发**：如果你的 AI 工具支持自定义记忆功能，最好设置自动整理策略，而不是完全靠手动维护。

### 3. 记忆内容有严格的大小限制

MEMORY.md 每行 150 字符，总行数也有限制——这是刻意为之的工程约束。

目的是确保记忆信息不会成为新的"上下文炸弹"。

**工程启发**：做记忆系统时，限制一定要提前定好，否则记忆本身会变成负担。

## 写给想深度用好 AI 的人

三层记忆系统背后有一个核心认知转变：

**不要把 AI 当成"一次性的问答机器"，而要把它当成"逐渐了解你的项目和习惯的长期助手"。**

大多数人对 AI 的期待是：我说清楚，它做正确。但 AI 的记忆是短的，它记不住长对话以外的东西。

三层记忆系统告诉你：**如果你不给 AI 建立记忆机制，它就真的不会记得。**

这不是 AI 的缺陷，是现实。你要么接受每次重新解释，要么建立系统，让 AI 记住。

后者，才是 AI 编程的正确打开方式。
