## 先说一个让人汗颜的真相

用 Claude Code 三个月了，你可能只用过它的"提问-回答"模式。

但 Claude Code 实际上有 **50+ 个斜杠命令**（slash commands），覆盖从代码提交到环境诊断、从项目审查到测试生成的完整开发工作流。

这不是功能堆砌。看完第二层源码之后我发现，这些命令背后有统一的设计理念：**每个命令，其实都是给 AI 发的一个结构化指令。**

理解了这个，你就不只是"用命令"，而是真正理解了"如何让 AI 精准执行任务"。

## 为什么斜杠命令比自然语言更强大

在用斜杠命令之前，你可能一直这样做：

> "帮我看看这个项目有没有什么问题"
> "能不能给我写一个提交信息"
> "检查一下代码风格有没有问题"

这些自然语言指令有一个共同问题：**模糊性**。

"看看有没有问题"——什么问题？代码风格？逻辑错误？性能隐患？
"写一个提交信息"——要中文还是英文？要详细还是简洁？

斜杠命令解决的正是这个问题：**把模糊意图变成清晰指令**。

当你输入 `/review`，Claude Code 知道要执行一套标准的代码审查流程——读取变更 → 分析逻辑 → 检查风格 → 输出报告。你不需要每次都解释"代码审查要包含哪些步骤"。

## 核心命令详解

### /commit：让 AI 替你写提交信息

这是被使用最多的命令之一，但 90% 的人用法不对。

**错误用法：**
```
/commit
```

**正确用法：**
```
/commit --all
```

或者先 `git add`，然后：
```
/commit
```

**背后的原理：**

Claude Code 的 `/commit` 不是简单调用 `git commit`，而是：
1. 读取 `git diff` 获取变更内容
2. 分析变更的意图（是 bug 修复？新功能？还是重构？）
3. 根据 Angular 规范生成符合约定的提交信息
4. 自动推荐 `git push` 时要推送哪些关联提交

**进阶技巧：**

```
/commit --verbose
```
加上 `--verbose`，Claude Code 会输出它对这次变更的理解，你可以纠正它的判断。相当于"AI 起草，人来审核"。

---

### /review：系统性代码审查

这是最能体现斜杠命令价值的场景。

**执行方式：**
```
/review
```

Claude Code 会：
1. 获取当前分支与主分支的 diff
2. 逐文件分析变更逻辑
3. 识别潜在问题：逻辑错误、边界条件、安全隐患、性能问题
4. 按严重程度分级输出

**但关键在于：它不是一次性输出所有问题，而是逐文件推进。**

源码中的审查流程是**流式输出**——先给出发现，立刻暂停，等待你确认"继续"再分析下一个文件。这意味着你可以随时介入，对某个具体问题深入追问。

**一个真实场景：**

代码里有个 `if (user.isAdmin)` 的判断。Claude Code 在 `/review` 中标记了这个问题：

> ⚠️ 权限判断依赖单一字段，建议增加审计日志

你可以立刻追问：
> "这个判断在其他地方有没有出现过？帮我搜一下"

这就是斜杠命令 + 追问的组合威力——审查发现了问题，你不需要重新解释上下文，直接追问。

---

### /doctor：你的环境保险单

很多人不知道这个命令有多实用。

**执行：**
```
/doctor
```

Claude Code 会检查：
- Node.js / Python / Bun 运行时版本是否匹配项目要求
- 关键依赖是否已安装（`node_modules` 是否完整）
- Git 配置是否正常
- 环境变量文件 `.env` 是否存在且格式正确
- 不同系统之间的潜在兼容性问题（尤其 macOS 和 Linux）

**它输出的是"为什么项目在我电脑上跑不起来"的完整诊断报告。**

比如报错 `Module not found`，你直接问 AI "这个错误怎么解决"，AI 可能会给出一堆通用建议。但如果先跑一遍 `/doctor`，它会告诉你具体是哪个包的哪个版本缺失了，以及应该怎么安装。

---

### /test：不只是生成测试用例

```
/test
```

很多人以为 `/test` 只是"帮我写测试"。但它的实际行为是：

1. 分析代码的边界条件和分支路径
2. 识别应该覆盖但当前没有覆盖的测试场景
3. 生成测试用例（不是随便写几个 assert 了事）
4. **运行测试，验证覆盖率**

重点是第 4 步——它会实际执行测试，确认没有引入新的错误。

**一个典型场景：**

你刚写了一个排序函数，跑 `/test`，Claude Code 发现：
- 已有测试覆盖了升序和降序
- **边界情况：空数组、只有一个元素、全部相同元素** 这三个场景没有覆盖
- 它会先补全这三个测试，运行，通过后才停下

---

### /diff：比 git diff 更智能的变更查看

```
/diff
```

这不只是展示变更内容，而是：
- 分析每处变更的**意图**（为什么改这里？）
- 标记可能引入问题的变更
- 提示相关的未同步修改（比如改了 A 文件但 B 文件引用没更新）

比 `git diff` 输出的信息密度高得多，但比完整代码审查更快。

适合在提交前快速扫一眼"有没有漏的东西"。

---

### /grep：跨文件的精准搜索

```
/grep "pattern"
```

不是简单的字符串匹配，而是**语义搜索**：

```
/grep "权限验证"
```

Claude Code 会理解你在找"与权限验证相关的代码"，而不是搜"权限验证"这个精确字符串——`hasPermission`、`checkAccess`、`isAuthorized` 这些变体都会被找到。

**结合 `/review` 使用的工作流：**

```
/review  →  发现某个模块有潜在问题
/grep "那个模块的名字" → 精确定位所有相关代码
```

## 命令的组合工作流

斜杠命令的真正威力，来自**组合使用**。

### 工作流一：提交前检查（黄金三步）

```
/diff        # 快速看变更
/review      # 系统性审查
/commit     # 生成提交信息
```

```
顺序执行下来的结果：
第一步，你确认了"改了什么"
第二步，Claude Code 找出了潜在问题
第三步，在确认没有问题之后，生成规范的提交信息
```

### 工作流二：接手新项目

```
/doctor      # 诊断环境是否就绪
/cat CLAUDE.md  # 查看项目说明（如果没有，先创建）
/grep "核心业务逻辑"  # 快速了解关键代码位置
```

```
顺序执行下来的结果：
第一步，确保环境没问题
第二步，了解项目背景和约定
第三步，找到核心代码的位置
```

### 工作流三：修复 Bug

```
/grep "出错的函数名"  # 定位相关代码
/test  # 先跑测试看是否有现成测试失败
# 告诉 Claude Code 你观察到的错误行为
# Claude Code 结合上下文理解问题，给出修复方案
```

## 斜杠命令的设计哲学：结构化优于模糊

看完全部命令之后，我发现它们有一个共同的设计原则：

**每个命令都对应一个固定的"做事套路"**——固定的意思不是死板，而是有明确的目标和流程。

```
/commit  →  分析变更 → 生成规范信息 → 确认 → 执行
/review  →  获取变更 → 逐文件分析 → 分级输出 → 等待追问
/test    →  分析覆盖 → 补全场景 → 运行验证 → 报告结果
```

当我们用自然语言指挥 AI 时，这个"流程"需要 AI 自己推断——所以经常出现"AI 跑偏了"的情况。

斜杠命令把流程固定下来，AI 的精力就可以集中在**执行质量**上，而不是**理解要做什么**上。

**对普通用户的启示：**

这个原则不只适用于 Claude Code。任何 AI 任务中，**把模糊意图翻译成结构化指令**都能大幅提升输出质量。

不要只说"帮我看看代码"，而是说"帮我检查三个地方：1. 权限判断是否有漏洞 2. 错误处理是否完整 3. 是否有潜在的空指针异常"。

结构化的提问 = 结构化的回答。

## 命令速查表

| 命令 | 核心功能 | 使用场景 |
|---|---|---|
| `/commit` | 生成提交信息并提交 | 准备提交代码时 |
| `/review` | 系统性代码审查 | 提交前检查、Code Review |
| `/doctor` | 环境诊断 | 项目跑不起来时 |
| `/test` | 生成并运行测试 | 写完代码后验证 |
| `/diff` | 智能变更查看 | 提交前快速确认 |
| `/grep` | 语义化代码搜索 | 定位问题时找相关代码 |
| `/vim` | 集成 Vim 操作 | 在编辑器中处理复杂编辑 |
| `/web` | 联网搜索 | 查文档、查错误信息 |
| `/claude` | 查看 AI 的思考过程 | 调试 AI 的行为 |
