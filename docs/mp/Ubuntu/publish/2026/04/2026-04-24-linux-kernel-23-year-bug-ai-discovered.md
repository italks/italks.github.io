## 一个 Bash 脚本，一个周末，23 年的内核漏洞

2026年4月12日，Linus Torvalds 正式发布了 **Linux 7.0** 内核。

全球开发者都在欢呼 Rust 转正、性能暴涨、硬件支持全面升级。但在这场狂欢背后，藏着一个让所有 Linux 运维人员后背发凉的故事：

**有人用一个简单的 Bash 脚本 + AI，挖出了一个藏在内核里整整 23 年的高危漏洞。**

而这个漏洞，影响的可能是你公司的文件服务器、医院的存储系统、云上的实例……

---

## 🎯 漏洞到底是什么？

### CVE-2026-31402：NFSv4 里的"定时炸弹"

| 属性 | 详情 |
|:---|:---|
| **漏洞编号** | CVE-2026-31402 |
| **位置** | Linux 网络文件共享代码 **NFSv4** |
| **类型** | 内存溢出类漏洞 |
| **根源代码提交时间** | 2003 年 3 月（Git 还没诞生！） |
| **潜伏时长** | **23 年** |
| **危害等级** | 高危（可接管整台文件服务器） |

### 谁发现的？怎么发现的？

发现者叫 **Nicholas Carlini**，Google DeepMind 的安全研究员。

他的方法出人意料地简单：

```bash
# 核心逻辑（简化版）
for file in linux/kernel/source/files/*.c; do
    cat $file | claude-opus-4.6 \
        --prompt "假装你在打CTF夺旗赛，你能搞坏什么？"
done
```

没错，就是遍历内核源码，把每个文件扔给 **Claude Opus 4.6**，然后问它一个极其直白的问题：

> **"如果这是 CTF 夺旗赛目标，你会怎么攻击它？"**

### 发现过程：从噪音到震撼

Carlini 的脚本其实跑了 **好几个月**，返回的大部分都是噪音和误报。

直到某个 **周五晚上**，脚本自动运行后突然停了下来——返回了一个真正的高危结果。

Carlini 本人坦言：

> "我这辈子从没发现过这种东西……这非常、非常、非常难。"

但讽刺的是，**发现这个 23 年老漏洞的"武器"，本身并不聪明**——只是算力成本低廉到一个人一周的预算就能跑起来。

- 成功利用后：**可接管整台文件服务器**，查看 HR 工资表等敏感数据

## 🤖 AI 找 Bug：从"垃圾"到"新常态"

### 这不是个例，是趋势

Linux 7.0 的发布标志着一个技术转折点：**AI 正式成为寻找深层次内核漏洞的有效武器**。

来看看数据：

#### 内核维护者的态度转变

| 时间段 | AI 提交的漏洞报告质量 | 维护者反应 |
|:---|:---|:---|
| 2025 年及以前 | 行号错误、无法复现、全是噪音 | 被称为 **"AI 垃圾"**，直接忽略 |
| 2026 年 2 月起 | 精确行号 + 可复现步骤 + 通顺根因分析 | Greg K-H 确认收到了**"真报告"** |

Greg Kroah-Hartman（Linux 稳定版分支维护者）确认：从 2026 年 2 月底开始，收到的 AI 提交报告**质量发生了质的飞跃**。

#### Linus 怎么看？

在 Linux 7.0 的发布邮件中，Linus Torvalds 亲口说：

> "我猜会有大量 AI 工具继续帮我们找各种边角 case，**这至少一段时间内会是新常态**。"

注意措辞——不是"可能"，是"**会**"。Linux 之父已经接受了现实。

### Google 的 Sashiko：AI 审查补丁工具

Google 工程师 Roman Gushchin 推出了名为 **Sashiko** 的 AI 审查工具（Rust 编写）：

- 它阅读 **LKML（Linux 内核邮件列表）**上的每一个补丁
- 在基于真实开发补丁的基准测试中：
  - **捕捉住了 53% 后来被确认为 Bug 的补丁**
  - 这些 **全都是人类审阅者漏掉的**

```
人类审阅漏掉的 Bug ──────── Sashiko（AI）捕获了 53%
        ↓
   进入生产环境 → 被用户触发 → 报告 → 修复
        ↓
              Sashiko 在代码合并前就拦下来了 ✅
```

## 🔧 你现在该做什么？

### 行动清单

#### 1️⃣ NFS 用户：立即检查！

```bash
# 检查当前内核版本
uname -r

# 检查是否已安装 CVE-2026-31402 补丁
# （需要升级到已修复此漏洞的内核版本）
cat /proc/version

# 如果使用 Ubuntu 26.04 LTS（刚发布！），请确保：
sudo apt update && sudo apt full-upgrade
```

**Ubuntu 26.04 LTS 用户请注意**：刚发布的版本搭载 Linux 7.0 内核，应已包含此修复。但如果你是旧版本用户，**务必立即更新**。

#### 2️⃣ XFS 用户：关注自愈功能

Linux 7.0 引入了新的 `xfs_healer` 自愈守护进程，可以自动修复 XFS 元数据损坏，减少停机风险。如果你使用 XFS 文件系统，值得关注。

#### 3️⃣ 开发团队：审查遗留代码

这个 23 年的老漏洞给所有人的警示：

> **那些编写于 2010 年以前、原作者早已离职但仍在运行的遗留代码，现在面临被 AI 工具快速扫描出类似高危漏洞的风险。**

建议行动：
- 盘点仓库中的老旧 C/C++ 代码模块
- 使用 AI 辅助代码审计工具进行一轮扫描
- 对核心基础设施代码建立定期审查机制

## 📊 Linux 7.0 安全相关更新一览

除了这个重磅漏洞发现，Linux 7.0 还包含多项安全加固：

| 安全特性 | 说明 | 影响用户 |
|:---|:---|:---|

| **通用 I/O 错误报告框架** | 统一错误处理机制 | 存储系统开发者 |
| **CVE-2026-31402 修复** | NFSv4 溢出漏洞补丁 | **所有 NFS 用户（必看！）** |

## 🧠 深度思考：AI 改变了什么？

### 三层冲击波

**第一层：安全范式转移**

过去：靠人类专家的经验和直觉找漏洞 → 成本高、覆盖面有限
现在：AI + 廉价算力 = 任何人都可能挖出深埋多年的 bug

**第二层：代码资产重新估值**

你仓库里那堆"一直能用就没动过"的遗留代码？在 AI 面前可能**脆弱得像纸一样**。23年的代码从未被发现有问题，不是因为安全，而是因为**没人认真看过**。

**第三层：攻防平衡被打破**

当发现漏洞的成本降到一个人一周的预算时：
- 白帽子会用 AI 找 bug
- 黑帽子也会
- **窗口期会被大幅压缩**

### 对 Ubuntu 用户的实际意义

好消息是：**Ubuntu 26.04 LTS 昨天刚发布，搭载的就是修复了这些问题的 Linux 7.0 内核。**

如果你还在犹豫要不要升级，这个漏洞故事应该能帮你下定决心：

> **新版本不仅更快更好——它还堵上了一个 23 年的老窟窿。**
